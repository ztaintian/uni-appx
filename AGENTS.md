# AGENTS.md

本文件用于沉淀个人 uni-app 项目的通用开发约定。后续新建或维护 uni-app 项目时，优先按本文件要求组织项目架构、页面、组件、接口和公共逻辑。

## 总体原则

- 默认使用 Vue 3 和组合式写法，页面和组件优先采用 `<script setup>`。
- 项目结构要清晰分层，页面只负责页面状态、页面组装和页面级交互。
- 复用 UI 必须组件化，公共逻辑必须方法化，接口请求必须统一封装。
- 不要把页面写成一个超大文件，也不要在页面里堆业务常量、模拟数据和请求细节。
- 能沉淀为公共模块的内容，应优先放到对应目录中复用。
- 修改代码时保持范围克制，不做与当前需求无关的大规模重构。

## 推荐目录结构

```text
.
├── api/                  # 接口模块，按业务域拆分
├── components/           # 全局公共组件
├── composables/          # 组合式函数，封装可复用状态和业务逻辑
├── constants/            # 常量、枚举、字典配置
├── pages/                # 页面文件
├── static/               # 静态资源
├── stores/               # 全局状态管理
├── styles/               # 公共样式、主题变量、mixin
├── utils/                # 工具方法
├── pages.json            # 页面和导航配置
├── manifest.json         # 应用配置
└── App.vue               # 应用入口
```

如果项目规模较小，可以先保留 `components/`、`utils/`、`api/`、`constants/` 等核心目录；如果后续复杂度增加，再补充 `stores/`、`composables/`、`styles/`。

## 页面规范

- 页面放在 `pages/` 下，并同步注册到 `pages.json`。
- 页面文件优先使用 `<script setup>`。
- 页面只处理：
  - 页面状态。
  - 页面生命周期。
  - 页面级事件。
  - 组件组装。
  - 页面跳转。
- 页面不直接写复杂请求逻辑，请求应放到 `api/`。
- 页面不直接写复杂数据转换，转换应放到 `utils/` 或 `composables/`。
- 页面不直接写大量静态配置，配置应放到 `constants/`。

## 组件规范

- 可复用 UI 放到 `components/`。
- 组件命名使用清晰语义，例如 `BaseButton.vue`、`AppSearchBar.vue`、`JobCard.vue`。
- 公共基础组件可使用 `Base` 前缀。
- 业务组件可按业务语义命名。
- 组件通过 `props` 接收数据，通过 `emit` 通知父级。
- 组件内部不要直接修改父级状态。
- 组件不要强依赖某个页面的私有数据结构，必要时通过 props 适配。
- 列表项、卡片、筛选栏、搜索栏、空状态、加载状态、底部导航等优先拆成组件。

## 组合式函数规范

- 可复用状态和业务逻辑放到 `composables/`。
- 文件命名使用 `useXxx.js` 或 `useXxx.ts`。
- 适合放入 `composables/` 的内容：
  - 列表分页逻辑。
  - 搜索和筛选状态。
  - 表单状态和校验。
  - 上传流程。
  - 定位、权限、设备能力调用。
  - 页面间可复用的业务流程。
- `composables/` 可以调用 `api/` 和 `utils/`，但不要混入大量 UI 模板逻辑。

## 接口封装规范

- 所有接口请求统一放到 `api/`。
- 请求基础能力统一封装，例如 `utils/request.js`。
- 页面不直接调用 `uni.request`。
- 接口模块按业务域拆分，例如：
  - `api/user.js`
  - `api/auth.js`
  - `api/order.js`
  - `api/upload.js`
- 请求封装应统一处理：
  - baseURL。
  - token。
  - header。
  - loading 策略。
  - 错误提示。
  - 登录失效。
  - 返回数据格式。
- 上传、下载、文件预览等能力也应统一封装，不要散落在页面里。

## 状态管理规范

- 简单页面状态使用 `ref`、`reactive`、`computed`。
- 多页面共享状态放到 `stores/`。
- 用户信息、token、主题、全局配置、购物车、权限等适合放入全局状态。
- 不要滥用全局状态；只在跨页面共享时使用。
- 本地缓存读写应封装到工具方法，避免页面中到处直接调用 `uni.getStorageSync` 和 `uni.setStorageSync`。

## 常量和配置

- 常量、枚举、字典放到 `constants/`。
- 不要在页面和组件中散落魔法字符串、状态码、业务类型值。
- 路由路径、Tab 配置、筛选项、状态字典等可按复杂度放到 `constants/` 或独立配置文件。
- 模拟数据如果只是开发阶段使用，应集中放置，避免混在页面逻辑里。

## 工具方法规范

- 通用工具方法放到 `utils/`。
- 工具方法应尽量保持纯函数，输入明确、输出明确。
- 适合放到 `utils/` 的内容：
  - 日期格式化。
  - 金额格式化。
  - 字符串处理。
  - 数据映射。
  - 表单校验。
  - storage 封装。
  - request 封装。
  - 路由跳转封装。
- 不要把强业务状态写进通用工具方法里。

## 样式规范

- Vue 文件中的样式默认使用 `<style lang="scss" scoped>`。
- `uni.scss` 保留为 uni-app 约定入口文件，用于引入 `styles/` 下的公共样式文件。
- 主题变量放到 `styles/theme.scss`。
- 公共 mixin 放到 `styles/mixins.scss`。
- 公共样式类放到 `styles/common.scss`。
- 页面样式尽量只写当前页面布局。
- 组件样式尽量使用 `scoped`，避免污染全局。
- 移动端尺寸优先使用 `rpx`。
- 固定格式 UI 要有稳定宽高或最小尺寸，避免内容变化导致布局跳动。
- 页面整体应保持移动端优先，信息清晰，操作区域足够大。
- 不要过度装饰，不要为了视觉效果牺牲可读性和可维护性。

## 导航和路由

- 新增页面必须同步维护 `pages.json`。
- TabBar 或自定义底部导航应集中维护配置。
- 页面跳转路径不要散落在各处，复杂项目应封装路由工具。
- 跳转前如果需要登录、权限或参数校验，应统一封装。
- 自定义底部导航必须保证每个菜单对应真实页面，不要只做静态按钮。

## 平台兼容

- 涉及 App、小程序、H5 差异时，使用 uni-app 条件编译。
- 平台专属能力要隔离封装，不要直接散落在业务页面中。
- 调用相机、定位、相册、文件、推送等能力时，要考虑权限失败和用户拒绝场景。
- 不确定平台行为时，优先写清楚兼容边界。

## 错误处理和空状态

- 列表页应考虑加载中、空数据、请求失败、没有更多等状态。
- 表单页应考虑校验错误、提交中、提交失败、重复提交。
- 接口错误应统一处理，页面可按业务需要补充提示。
- 用户可感知的失败要有明确反馈，不要静默失败。

## 开发和验证

- 修改前先看现有目录和代码风格，再按已有模式扩展。
- 完成后至少检查：
  - 页面是否已注册到 `pages.json`。
  - import 路径是否存在。
  - Vue 单文件组件结构是否完整。
  - 组件 props 和 emit 是否匹配。
  - 页面中文是否正常显示。
  - 底部导航或页面跳转是否对应真实页面。
- 如果项目没有 npm 脚本，则使用 HBuilderX 运行和预览。
- 如果存在构建、lint、类型检查或测试脚本，修改后优先运行相关脚本。

## 生成代码时的默认要求

- 用户要求生成页面、功能或 App 时，优先直接实现。
- 默认按页面、组件、接口、工具方法、常量配置分层。
- 不要只给单文件大页面，除非用户明确要求快速演示。
- 不要留下无用的示例代码和大段注释。
- 如果新增功能需要模拟数据，应集中放到数据或常量文件中。
- 如果后续容易接接口，应提前留出 `api/` 和数据转换边界。
